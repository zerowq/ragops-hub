# RAGOps Hub 项目面试问答

答案按口语表达编写，面试时不需要逐字背诵，记住每题的“设计原因、验证方式、当前边界”即可。

## 一、项目介绍

### 1. 你先介绍一下这个项目

我做的是一个多租户企业知识库和客服 Agent。文档上传后会经过解析、结构化切分和 Embedding，
Dense 走 Milvus，Sparse 走 FTS5 倒排索引，然后通过 RRF 融合和轻量重排。Agent 会区分知识问答、
订单查询和创建工单，创建工单必须让用户确认，并且有幂等和审计。

### 2. 它和普通 RAG Demo 最大的区别是什么

普通 Demo 通常只验证“能不能回答”，我更关心企业里“能不能安全、稳定地回答”。所以项目重点不是
聊天页面，而是租户权限、工具边界、引用、幂等、审计、评测和故障处理。

### 3. 这算不算 Agent

我会把它叫作受控的 Agent Workflow。它会根据意图选择 RAG 或工具，也有会话状态和人工确认，但
不会让模型任意规划很多步骤。这是我故意做的取舍，因为客服系统里可解释和可审计比完全自主更重要。

### 4. 为什么没有使用 LangChain 或 LangGraph

项目使用了 FastAPI、Milvus、PyMilvus、Pydantic、HTTPX、PyPDF 和 python-docx 等开源组件，但
RAG 编排、RRF、轻量重排、权限规则和 Agent 状态流是自己实现的。这样做是为了把核心机制写清楚，
不是认为框架没用。流程继续复杂以后，我会考虑 LangGraph，但权限、幂等和审计仍然要自己设计。

### 5. 全部自己实现，面试官会不会追问细节

会，而且这其实是项目的优势。要准备好讲 tokenize、BM25/FTS5、RRF、Milvus Filter、Chunk、
Pending Action、幂等键和 SSE 事件。不要只说“我调了一个框架接口”，要能顺着一次请求讲到数据库
和向量库。

## 二、检索与 RAG

### 6. 为什么用 Milvus

Milvus 在这个项目里负责高维向量索引和相似度搜索。小数据用内存或 pgvector 也可以，但数据规模、
并发和索引调优要求上来后，独立向量服务更合适。我通过 VectorStore 接口隔离了业务和具体 SDK。

### 7. 用了 Milvus，为什么 BM25 之前还会全量扫描

因为原来的 Milvus Collection 只有 Dense 的 `FLOAT_VECTOR`，并没有配置 Sparse Vector 和 BM25
Function。原 BM25 是应用层把 SQLite Chunk 全部读出来临时计算的。本轮已经改成 SQLite FTS5
持久倒排索引，所以查询不再全量重建 BM25。

### 8. 为什么不直接加 Elasticsearch/OpenSearch

它当然可以，而且复杂中文分词、同义词和字段权重方面很成熟。但这个项目强调一键部署，如果再加
OpenSearch，会多一个重服务和一套双写问题。当前用 FTS5 解决本地扫描问题；生产可以根据规模选择
Milvus 原生 BM25 或 OpenSearch。

### 9. 为什么需要 Hybrid Retrieval

Dense 擅长同义表达，比如“怎么退钱”和“退款流程”；Sparse 擅长产品名、订单号、错误码这些精确
词。企业文档两种内容都有，所以混合召回通常比单路稳定。

### 10. RRF 为什么不用两个原始分数直接相加

Dense 相似度和 BM25 分数的分布不是一个量纲，直接相加需要做复杂校准。RRF 只看各自排名，
`1 / (k + rank)`，两边都靠前的候选自然会得到更高分。

### 11. 为什么 RRF 后面还要轻量重排

真实评测时有一条精确 BM25 结果没有进入 Dense 列表，被两个列表都出现的普通结果挤下去了。我在
RRF 后增加词项覆盖率和归一化 Sparse 分数，并加了回归测试。当前权重还是小数据基线，生产会用
Cross-Encoder 重新评测。

### 12. Top-K 怎么选

召回 K 和最终上下文 K 要分开。召回阶段可以多取一些，融合重排后只给模型 3～5 条。最终通过
Recall@K、MRR、答案质量、上下文 Token 和延迟一起决定，不是固定经验值。

### 13. Chunk Size 和 Overlap 怎么选

Chunk 太小会切断语义，太大会让检索不精确，还浪费上下文。现在 500/80 是演示基线，项目会对
标题、段落、句子和长文本逐级切分，并保证 Chunk 不超过上限。生产要用固定评测集比较不同参数。

### 14. 知识库没有答案时怎么办

不能因为向量库总能返回 Top-K 就强行回答。本轮增加了 Dense 和词项证据门槛，候选不够可靠就
拒答。后续还要在评测集中专门统计正确拒答率和错误回答率。

### 15. Citation 能证明答案一定正确吗

不能。Citation 只能说明系统用了哪些来源，不代表模型对来源的理解一定正确。还需要做答案忠实度、
引用准确率和人工抽查，最好把回答中的关键 claim 和引用片段再做一次校验。

## 三、权限与安全

### 16. 为什么权限必须在检索前过滤

如果先取 Top-K 再过滤，无权数据可能已经占满候选位，既影响召回，也增加泄漏风险。Milvus Dense
直接用 tenant、department、owner 做 Filter；FTS5 查询也在 SQL 中带同样的 ACL 条件。

### 17. 当前身份系统怎么设计

演示模式可以通过 Header 切换用户，方便现场展示。生产模式设置 `AUTH_MODE=jwt`，服务端从签名
Token 的 claims 构造 Principal，客户端不能直接决定 tenant 和 user。真实企业环境会进一步接
OIDC/JWKS。

### 18. 私有文档如何防止别人看到或删除

Document 和 Chunk 都保存 owner。列表、Sparse 检索、Dense 检索结果校验和删除都使用同一套
tenant/department/owner 规则。删除还要求是 owner 或知识库管理员。

### 19. Prompt Injection 怎么防

正则只是第一层，能挡明显攻击，但不是完整方案。项目还把检索内容标记成不可信数据、过滤高风险
Chunk、让 RAG 分支不能直接执行写工具，并通过权限和人工确认限制攻击后果。我的表述会是“分层
降低风险”，不会说完全解决。

### 20. 知识库文档自己带恶意指令怎么办

这是间接 Prompt Injection。本轮在检索后会过滤命中高风险规则的 Chunk，Prompt 也明确说明上下文
只是数据，不是指令。更完整的方案还包括入库扫描、来源信任等级、隐藏文本检测和专门分类模型。

## 四、Agent 与工具

### 21. 为什么 Agent 不能直接访问数据库

模型输出不能当成可信命令。模型最多负责识别意图和生成结构化参数，真正的工具还要做参数校验、
对象权限、超时、幂等和审计，数据库凭证也不能交给模型。

### 22. 订单查询如何防止水平越权

查询条件不是只有订单号，而是 `order_id + tenant_id + user_id`。用户即使猜到别人的订单号，工具
内部也会因为 owner 不匹配而拒绝。

### 23. 为什么创建工单必须二次确认

订单查询是读操作，创建工单是有副作用的写操作，可能触发客服流程和 SLA。系统会先展示将要执行
的内容，用户明确确认后才写入。

### 24. 幂等怎么实现

每个待确认动作都有服务端生成的 `action_id`，幂等键由 tenant、user 和 action_id 计算，数据库还有
唯一约束。工单写入使用原子 `INSERT OR IGNORE`，多次网络重试会返回同一张工单。

### 25. 用户确认时服务挂了怎么办

Pending Action 不会在执行前直接删除。工单写成功后才按 action_id 清理，失败时仍可重试。更复杂的
外部工单系统可以把动作做成 prepared、executing、succeeded、failed 状态机。

### 26. conversation_id 被别人猜到怎么办

更新和读取会话时必须同时匹配 tenant 和 user。相同 ID 如果已经属于另一个 Principal，会直接拒绝，
不能覆盖对方的 Pending Action。生产还会优先使用服务端生成的随机会话 ID。

### 27. 为什么 Intent Router 不直接用大模型

当前意图少而且边界明确，规则路由便宜、快、可测试。它也处理了创建工单否定词和“订单退款政策”
这类歧义。意图继续增加后可以用小分类模型，但低置信度时仍要澄清，不能直接执行写工具。

## 五、工程与部署

### 28. 项目能不能一条命令启动

可以。执行 `docker compose --profile full up -d --build` 会启动 etcd、MinIO、Milvus、自动知识入库
任务和 FastAPI。Bootstrap 成功后应用才启动，健康检查通过后访问 8000 端口。

### 29. 为什么 Milvus 还需要 etcd 和 MinIO

Milvus Standalone 仍然把元数据交给 etcd，把持久对象数据放到 MinIO。Docker Compose 把这三个
服务编排在一起，并通过健康检查控制启动顺序。

### 30. 为什么文档入库不能一直放在 HTTP 请求里

PDF 解析、Embedding 和索引都可能很慢，放在请求里容易超时，也会占住 Worker。当前解析和 Milvus
调用已经避免直接阻塞事件循环，下一步生产化会把整个入库流程放到任务队列，接口只返回 processing。

### 31. 向量库和关系库双写失败怎么办

当前先有状态管理，并在元数据写失败时补偿删除向量；Dense 结果还会用关系库做存在性校验，孤立
向量不会进入最终结果。正式生产仍建议 Outbox、重试、删除状态和定时一致性巡检。

### 32. 为什么本地使用 SQLite

为了让项目没有 PostgreSQL 也能快速运行和测试。SQLite 现在还承担 FTS5 Sparse Index。生产多实例
部署会换 PostgreSQL，并根据规模把 Sparse 迁移到 Milvus BM25 或 OpenSearch。

### 33. 为什么使用 SSE，不用 WebSocket

这里主要是服务端持续往客户端推事件，双向实时通信需求不强，SSE 更简单。它还能定义检索、工具、
确认、Token、引用和结束等具名事件。语音或高频双向协作才更适合 WebSocket。

### 34. 健康检查检查什么

不只是返回配置。当前会实际检查 SQLite 和 Vector Store 是否可用，Docker 也用这个接口判断应用
健康。生产会进一步拆成 liveness 和 readiness，并加入模型服务和队列状态。

## 六、评测与项目边界

### 35. 为什么使用 Recall@K 和 MRR

Recall@K 看正确来源有没有进入候选，MRR 还关心它排第几。两者适合先评检索，但不代表答案一定
正确，所以还需要 Faithfulness、Citation Accuracy 和正确拒答率。

### 36. 5 道题都是 1.0，能不能说准确率 100%

不能。它只能说明这 5 道样例题的基线链路正确。数据太小、文档也简单，正式展示应该扩到至少
50～100 题，并加入无答案、干扰文档、越权和同义改写。

### 37. 7ms 检索延迟有代表性吗

只能算本机小数据基线。生产要看 P50、P95、P99，并拆开 Embedding、Dense、Sparse、Rerank 和生成
耗时，还要在真实并发和网络条件下压测。

### 38. 项目中最有价值的一次调优是什么

评测时发现一条强 Sparse 结果被融合候选挤下去。我没有直接换更大的模型，而是先看每题的两路排名，
再加词项覆盖和归一化 Sparse 信号，最后补回归测试。这个过程体现的是错误分析，而不只是调 API。

### 39. 当前项目最大的不足是什么

它仍是工程样板，不是大型生产系统。异步任务队列、OIDC、完整 Outbox、OCR、专业 Reranker、更大
评测集和线上观测还没有全部实现。我会明确区分“已经实现”和“生产演进方案”。

### 40. 如果数据变成一百万个 Chunk 怎么改

原文件放对象存储，元数据换 PostgreSQL，入库全部走 Worker。Dense 使用 Milvus，Sparse 根据团队
情况选择 Milvus BM25 或 OpenSearch，再做两阶段召回和 Cross-Encoder 重排，同时增加缓存、限流、
分区、监控和一致性任务。
